# Open Source Evolution: 印迹 → 通用建站工具

> 历史设计（2026-07-18 标记）：本文的共享数据库多站点章节保留用于追溯，不是当前实现目标。当前架构采用一个实例一个逻辑站点、域名别名由代理层处理、多个独立站点部署多个隔离实例。参见 `docs/adr/0001-single-instance-single-site.md`。

**日期**: 2026-03-11
**状态**: Draft
**目标**: 将印迹从企业官网 CMS 演进为通用开源建站工具，对标 Halo，建立社区生态

## 背景

印迹当前已实现：CMS 页面管理（8 固定页 + 动态页面）、草稿/发布/回滚、文章/博客、媒体库、表单收集、菜单管理、主题令牌系统（3 套主题包）、JWT 认证、审计日志、备份、双语内容管理。

参考 Halo（37k+ Stars，100+ 插件）的核心能力差距：插件化架构、应用市场、评论系统、全文搜索、AI 集成、多用户体系、定时发布、数据迁移等。

## 演进策略

**方案 B：功能先行，渐进插件化**

先快速补齐高价值功能，在开发过程中逐步抽象出插件接口，最终形成完整插件生态。

理由：
- 现有 CMS 功能基础不值得推倒重来
- 社区建设需要"能用的产品"吸引第一批用户
- 在实际开发中验证插件接口设计，避免过度抽象
- Go 的插件机制需要在实践中选型验证

技术栈完全开放，不排斥引入新技术。资源模型面向社区建设。

## 已知限制

- **双语而非多语**：架构目前仅支持 zh/en 双语。通用化为 N 语言不在本路线图范围内，但 Provider 接口设计时应预留 locale 参数的扩展性。

## 关键风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Go 插件运行时选型 | Phase 3 核心依赖 | 3.1.4/3.1.5/3.1.5b 三方案并行 POC 对比（含 HashiCorp go-plugin），提前 1 周决策 |
| 多站点数据隔离复杂度 | 所有查询需改造 | 使用 GORM scope 全局注入 site_id 过滤 |
| 社区冷启动 | 没有贡献者 | Phase 1 交付可用产品吸引用户，Phase 2 降低参与门槛 |
| AI 功能依赖外部 API | 成本和延迟 | Provider 接口解耦，支持自部署模型（Ollama） |
| SPA SEO 限制 | 搜索引擎/社交平台无法抓取动态内容 | Phase 0 引入后端模板注入 index.html meta 标签 |
| 多站点改造影响面大 | Phase 1-3 新增表均需追加 site_id | 设计表结构时预留 site_id 字段（nullable），Phase 4 激活 |

## 设计原则

1. **每个 Phase 独立可交付** — 即使后续 Phase 未完成，当前阶段的功能也有独立价值
2. **接口先行** — Phase 1 功能开发时就定义好扩展接口，Phase 2 抽象时水到渠成
3. **插件吃自己的狗粮** — Phase 3 的官方插件必须通过 SDK 开发，验证 SDK 完整性
4. **向后兼容** — 每个 Phase 不破坏已有功能和 API，通过版本化管理变更
5. **社区友好** — 代码、文档、工具都面向外部贡献者设计，不假设内部知识
